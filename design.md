# Delhi Scheme Saathi - Design Document

## Design Decision Summary

This section highlights key architectural decisions and their rationales, addressing requirements from the Requirements Document.

### 1. Voice-First Architecture (Requirements FR1)
**Decision:** Use Bhashini (AI4Bharat) as primary voice processing service with Google Cloud as fallback.

**Rationale:**
- Bhashini provides government-approved, India-hosted AI services
- IndicWhisper optimized for Indian languages and accents
- Data sovereignty compliance (Requirements NFR5)
- Fallback ensures 99.5% availability (Requirements NFR3)

### 2. Hybrid Scheme Retrieval (Requirements FR4)
**Decision:** Three-stage retrieval combining life event mapping, structured filtering, and semantic search.

**Rationale:**
- Pure semantic search misses hard eligibility constraints (age, income)
- Pure rule-based filtering lacks relevance ranking
- Hybrid approach achieves >80% precision and >70% recall targets
- Life event mapping bridges gap between natural language and structured data

### 3. Serverless Architecture (Requirements NFR1, NFR4)
**Decision:** AWS Lambda for compute, DynamoDB for sessions, PostgreSQL for schemes.

**Rationale:**
- Auto-scaling from 100 to 1000 concurrent users without manual intervention
- Pay-per-use cost model suitable for variable load
- Lambda cold start <1s acceptable for 5s latency budget
- DynamoDB TTL enables automatic data expiry (Requirements NFR2)

### 4. No Permanent Personal Data Storage (Requirements NFR2)
**Decision:** 24-hour TTL on all session data, no long-term user profiles.

**Rationale:**
- Privacy by design - minimize data breach risk
- Compliance with data minimization principles
- 24-hour window balances privacy with realistic user journeys (target users may take a day to gather documents before returning)
- Users can restart conversation anytime without privacy concerns
- Scheme data is public, personal data is ephemeral

### 5. Document Dependency Resolution (Requirements FR5)
**Decision:** Recursive depth-first traversal to resolve complete document chains.

**Rationale:**
- Users need to know prerequisites to get prerequisites (e.g., need Aadhaar to get Income Certificate)
- Prevents users from getting stuck mid-process
- Key differentiator from existing solutions (MyScheme, Jugalbandi)

### 6. Rejection Prevention System (Requirements FR6)
**Decision:** Proactive warnings before application submission.

**Rationale:**
- Applications rejected for minor errors cause user frustration
- Prevention is better than correction
- Increases success rate from ~8% baseline to 20% target
- Key differentiator addressing real user pain point

### 7. Conversation State Machine (Requirements FR9)
**Decision:** Explicit 10-state FSM with defined transitions.

**Rationale:**
- Prevents context loss across 20+ conversational turns
- Enables graceful error recovery
- Facilitates testing and debugging
- Clear separation of concerns for each conversation phase

### 8. Multi-LLM Fallback (Requirements NFR3)
**Decision:** Claude → GPT-4 → Gemini → Rule-based fallback chain.

**Rationale:**
- No single LLM provider guarantees 99.5% uptime
- Automatic failover maintains service availability
- Rule-based fallback for basic queries when all LLMs unavailable
- Cost optimization by using cheaper models as fallbacks

### 9. Anti-Hallucination Architecture (Requirements FR4, FR5, FR6)
**Decision:** Strict separation between LLM responsibilities (conversation) and rules engine responsibilities (scheme facts).

**Rationale:**
- LLM handles HOW: intent classification, entity extraction, natural language generation, conversational flow
- Rules engine handles WHAT: eligibility determination, document requirements, rejection rules, benefit amounts
- All scheme-specific data served from a curated, verified PostgreSQL database — never generated by the LLM
- If a query cannot be matched to a known scheme or document, the system explicitly states this and offers CSC handoff rather than generating a plausible-sounding answer
- This eliminates the class of failure where an LLM confidently states incorrect eligibility criteria or fabricates document requirements

### 10. Responsible AI Design
**Decision:** Structured eligibility filtering with full explainability, no opaque LLM-driven eligibility decisions.

**Rationale:**
- Eligibility is determined by SQL rules matching official government criteria (age, income, category), not by LLM inference — this prevents demographic bias in eligibility outcomes
- Every recommendation includes the specific criteria matched, so users understand WHY a scheme was recommended
- Rejection warnings cite specific rules and sources, not LLM-generated advice
- Voice transcription confidence scoring: low-confidence transcriptions trigger text fallback rather than acting on potentially misheard input
- Scheme data staleness tracking: data older than 90 days is flagged with a warning and official source link

### 11. Low-Bandwidth and Degraded Network Design (Requirements NFR3, NFR6)
**Decision:** WhatsApp-first delivery with progressive degradation across network conditions.

**Rationale:**
- WhatsApp works on 2G connections and compresses media automatically
- Voice messages use OGG Opus (8-16 kbps), allowing 30-second messages in ~60KB
- Text responses prioritized over audio when network quality is poor
- Cached scheme data in ElastiCache ensures core functionality during partial outages
- Rule-based fallback when external APIs are unreachable

## System Architecture Overview

Delhi Scheme Saathi is designed as a serverless, modular monolith system that provides voice-first assistance for Delhi government scheme navigation. The architecture prioritizes low latency, high availability, and seamless integration with government AI services.

**Architectural Pattern:** Modular monolith deployed as AWS Lambda functions with clear service boundaries. This approach provides the simplicity of a monolith with the modularity of microservices, avoiding distributed system complexity while maintaining code organization.

### Architecture Principles
- **Voice-First Design:** Optimized for Hindi voice interactions
- **Serverless Scalability:** Auto-scaling based on demand
- **100% Sovereign AI Stack:** All language processing through Bhashini (AI4Bharat, IIT Madras), all infrastructure in AWS Mumbai region
- **Privacy by Design:** No permanent personal data storage
- **Anti-Hallucination:** LLM handles conversation, rules engine handles scheme facts
- **Graceful Degradation:** Fallback mechanisms for all dependencies, down to text-only rule-based mode

- **Architecture Pattern:** Modular monolith with clear service boundaries
- **Concurrency:** Auto-scaling up to 1000 concurrent executions

### 4. External Language Services (Sovereign AI Stack)
- **Bhashini APIs (AI4Bharat, IIT Madras):**
  - IndicWhisper: Speech-to-text for Hindi
  - IndicTrans2: Hindi-English translation
  - IndicTTS: Text-to-speech for Hindi
- **Fallback:** Google Cloud Speech/Translate APIs
- **All Bhashini services are government-operated and hosted on Indian servers.**

### 5. External Reasoning Services
- **Primary LLM:** Claude 3.5 Sonnet (Anthropic)
- **Fallback:** GPT-4 (OpenAI) or Gemini Pro (Google)
- **Usage:** Intent classification, entity extraction, conversational response generation. LLM is never used to generate scheme facts, eligibility criteria, or document requirements — these are served from the curated database.

### 6. Data Layer
- **DynamoDB:** Session management, user profiles (24-hour TTL)
- **PostgreSQL + pgvector:** Schemes, documents, CSCs, embeddings
- **S3:** Audio files, static assets, document templates
- **ElastiCache:** Frequently accessed scheme data

**Session TTL Rationale:** 1-hour TTL (original design) was misaligned with actual user behavior. Target users — semi-literate residents navigating bureaucracy — may learn about a required document on Day 1, visit an SDM office on Day 2, and return to the chatbot on Day 2 to continue. A 24-hour TTL preserves context across these multi-day journeys while still ensuring automatic cleanup. All session data is still ephemeral and auto-deleted.

**Conversation Summary Rationale:** A 20+ turn conversation with a 10-message sliding window loses the user's initial situation description by the time they reach document guidance (turn 12-15). Every 5 turns, the LLM generates a compressed summary (~200 tokens) of the full conversation so far, which is prepended to the LLM context alongside the last 10 raw messages. This preserves early context without unbounded storage growth.

## Core Algorithms

### 1. Life Event Classification and Mapping

Maps user's natural language description to structured life events and associated schemes.

**Design Rationale:** Users don't know scheme names but can describe their situation. This algorithm bridges the gap between natural language and structured scheme discovery.

```python
class LifeEventClassifier:
    """
    Classifies user situation into life events using LLM-based intent classification.
    """
    
    # Life event taxonomy from Requirements FR2
    LIFE_EVENTS = {
        'JOB_LOSS': ['unemployment', 'job loss', 'laid off', 'company closed'],
        'PREGNANCY': ['pregnant', 'expecting', 'baby coming'],
        'CHILDBIRTH': ['baby born', 'newborn', 'delivery', 'child birth'],
        'MARRIAGE': ['wedding', 'getting married', 'shaadi'],
        'DISABILITY': ['accident', 'disabled', 'handicapped', 'injury'],
        'DEATH_IN_FAMILY': ['death', 'passed away', 'funeral'],
        'EDUCATION': ['school', 'college', 'study', 'scholarship'],
        'BUSINESS_STARTUP': ['business', 'startup', 'entrepreneur', 'shop'],
        'RETIREMENT': ['retired', 'pension', 'senior citizen'],
        'HOUSING': ['house', 'home', 'flat', 'accommodation'],
        'AGRICULTURE': ['farming', 'crops', 'agriculture']  # Limited for Delhi
    }
    
    async def classify_life_event(self, user_message: str, context: dict) -> str:
        """
        Classify user's situation into a life event category.
        Returns life event key or None if ambiguous.
        """
        # Use LLM for classification with structured output
        prompt = f"""
        User message: "{user_message}"
        
        Classify into ONE of these life events:
        {', '.join(self.LIFE_EVENTS.keys())}
        
        Return only the life event key or "UNCLEAR" if ambiguous.
        """
        
        life_event = await llm_classify(prompt)
        
        if life_event == "UNCLEAR":
            return None  # Trigger clarifying questions
        
        return life_event
```

### 2. Hybrid Scheme Retrieval Algorithm

The system uses a three-stage approach combining life event mapping, structured filtering, and semantic search for optimal relevance.

**Design Rationale:** Pure semantic search may miss eligibility constraints, while pure rule-based filtering lacks relevance ranking. Hybrid approach achieves >80% precision and >70% recall (Requirements FR4).

```python
def find_eligible_schemes(
    user_profile: UserProfile, 
    life_event: str,
    query: str
) -> List[Scheme]:
    """
    Three-stage hybrid retrieval:
    1. Life event filtering
    2. Structured eligibility filtering
    3. Semantic ranking using embeddings
    
    Validates: Requirements FR4 (Hybrid Scheme Retrieval)
    """
    
    # Stage 1: Life Event Filter
    life_event_schemes = db.query("""
        SELECT * FROM schemes 
        WHERE 
            :life_event = ANY(life_events) AND
            is_active = true
    """, {'life_event': life_event})
    
    # Stage 2: Structured Eligibility Filter (SQL)
    filtered_schemes = db.query("""
        SELECT * FROM schemes 
        WHERE 
            id = ANY(:scheme_ids) AND
            (min_age IS NULL OR min_age <= :age) AND
            (max_age IS NULL OR max_age >= :age) AND
            (max_income IS NULL OR max_income >= :income) AND
            (:category = ANY(categories) OR 'all' = ANY(categories)) AND
            (:gender = ANY(genders) OR 'all' = ANY(genders)) AND
            (NOT domicile_required OR :domicile = ANY(domicile_states))
    """, {
        'scheme_ids': [s.id for s in life_event_schemes],
        **user_profile.__dict__
    })
    
    # Stage 3: Semantic Ranking
    query_embedding = embed_text(query)
    
    for scheme in filtered_schemes:
        scheme.relevance_score = cosine_similarity(
            query_embedding, 
            scheme.description_embedding
        )
    
    # Stage 4: Sort and Return Top 5
    return sorted(
        filtered_schemes, 
        key=lambda s: s.relevance_score, 
        reverse=True
    )[:5]
```

### 2. Document Dependency Resolution

Resolves the complete chain of documents needed, including prerequisites.

**Design Rationale:** Users need to know not just what documents are required, but also what documents are needed to GET those documents. This recursive resolution prevents users from getting stuck (Requirements FR5).

```python
def get_document_chain(scheme: Scheme) -> List[Document]:
    """
    Get all documents needed for a scheme, including prerequisites.
    Uses depth-first traversal to resolve dependencies.
    
    Validates: Requirements FR5 (Document Procurement Guide)
    """
    all_documents = []
    visited = set()
    
    def add_with_prerequisites(doc_id: str):
        if doc_id in visited:
            return
        
        visited.add(doc_id)
        document = get_document(doc_id)
        
        # Add prerequisites first (depth-first)
        for prereq_id in document.prerequisites:
            add_with_prerequisites(prereq_id)
        
        all_documents.append(document)
    
    # Process all required documents
    for doc_id in scheme.documents_required:
        add_with_prerequisites(doc_id)
    
    return all_documents
```

### 3. Conversation State Management

Manages the flow of conversation through different states as defined in Requirements FR9.

**Design Rationale:** Structured state machine ensures consistent conversation flow and prevents context loss. Each state has clear entry/exit conditions and valid transitions.

**User Journey Mapping:** The user journey diagram shows 4 high-level stages that map to multiple conversation states:
- **Stage 1 (Language Processing):** Technical layer, no conversation state
- **Stage 2 (Understanding & Profiling):** GREETING → SITUATION_UNDERSTANDING → PROFILE_COLLECTION
- **Stage 3 (Scheme Matching):** SCHEME_MATCHING → SCHEME_PRESENTATION
- **Stage 4 (Actionable Output):** SCHEME_DETAILS → DOCUMENT_GUIDANCE → REJECTION_WARNINGS → APPLICATION_HELP → CSC_HANDOFF

```python
class ConversationStateMachine:
    """
    Manages conversation flow and state transitions.
    Implements 10 states from Requirements FR9.
    """
    
    # State definitions from Requirements FR9
    STATES = {
        'GREETING': 'Initial welcome and language detection',
        'SITUATION_UNDERSTANDING': 'User describes life event',
        'PROFILE_COLLECTION': 'Gathering eligibility attributes',
        'SCHEME_MATCHING': 'Running retrieval algorithm',
        'SCHEME_PRESENTATION': 'Showing eligible schemes',
        'SCHEME_DETAILS': 'Deep dive into specific scheme',
        'DOCUMENT_GUIDANCE': 'Explaining document requirements',
        'REJECTION_WARNINGS': 'Highlighting common mistakes',
        'APPLICATION_HELP': 'Step-by-step application guidance',
        'CSC_HANDOFF': 'Connecting to human help'
    }
    
    # Valid state transitions (includes backward transitions for natural conversation flow)
    TRANSITIONS = {
        'GREETING': ['SITUATION_UNDERSTANDING', 'PROFILE_COLLECTION'],
        'SITUATION_UNDERSTANDING': ['SITUATION_UNDERSTANDING', 'PROFILE_COLLECTION'],
        'PROFILE_COLLECTION': ['PROFILE_COLLECTION', 'SCHEME_MATCHING'],
        'SCHEME_MATCHING': ['SCHEME_PRESENTATION', 'CSC_HANDOFF'],
        'SCHEME_PRESENTATION': ['SCHEME_DETAILS', 'PROFILE_COLLECTION', 'CSC_HANDOFF'],
        'SCHEME_DETAILS': ['DOCUMENT_GUIDANCE', 'REJECTION_WARNINGS', 'APPLICATION_HELP', 'SCHEME_PRESENTATION'],
        'DOCUMENT_GUIDANCE': ['REJECTION_WARNINGS', 'APPLICATION_HELP', 'CSC_HANDOFF', 'SCHEME_DETAILS'],
        'REJECTION_WARNINGS': ['APPLICATION_HELP', 'CSC_HANDOFF', 'DOCUMENT_GUIDANCE', 'SCHEME_DETAILS'],
        'APPLICATION_HELP': ['CSC_HANDOFF', 'SCHEME_PRESENTATION', 'DOCUMENT_GUIDANCE'],
        'CSC_HANDOFF': ['SCHEME_PRESENTATION', 'GREETING']  # Non-terminal: user can return after CSC visit
    }
    
    def transition(self, current_state: str, user_input: str, context: dict) -> str:
        """
        Determine next state based on current state and user input.
        Validates: Requirements FR9 (Session Management)
        """
        # Classify user intent using LLM
        intent = classify_intent(user_input, context)
        
        # State transition logic
        if current_state == 'GREETING':
            if intent == 'describe_situation':
                return 'SITUATION_UNDERSTANDING'
            elif intent == 'ask_about_scheme':
                return 'PROFILE_COLLECTION'
        
        elif current_state == 'SITUATION_UNDERSTANDING':
            if intent == 'situation_clear' and context.get('life_event'):
                return 'PROFILE_COLLECTION'
            elif intent == 'need_clarification':
                return 'SITUATION_UNDERSTANDING'
        
        elif current_state == 'PROFILE_COLLECTION':
            if has_minimum_profile(context['user_profile']):
                return 'SCHEME_MATCHING'
            else:
                return 'PROFILE_COLLECTION'
        
        elif current_state == 'SCHEME_MATCHING':
            schemes = context.get('eligible_schemes', [])
            if len(schemes) > 0:
                return 'SCHEME_PRESENTATION'
            else:
                return 'CSC_HANDOFF'  # No schemes found, need human help
        
        elif current_state == 'SCHEME_PRESENTATION':
            if intent == 'learn_more_about_scheme':
                return 'SCHEME_DETAILS'
            elif intent == 'refine_profile':
                return 'PROFILE_COLLECTION'
            elif intent == 'need_human_help':
                return 'CSC_HANDOFF'
        
        elif current_state == 'SCHEME_DETAILS':
            if intent == 'ask_about_documents':
                return 'DOCUMENT_GUIDANCE'
            elif intent == 'ask_about_mistakes':
                return 'REJECTION_WARNINGS'
            elif intent == 'ask_how_to_apply':
                return 'APPLICATION_HELP'
        
        elif current_state == 'DOCUMENT_GUIDANCE':
            if intent == 'ask_about_mistakes':
                return 'REJECTION_WARNINGS'
            elif intent == 'ready_to_apply':
                return 'APPLICATION_HELP'
            elif intent == 'need_human_help':
                return 'CSC_HANDOFF'
            elif intent == 'go_back_to_scheme':
                return 'SCHEME_DETAILS'
        
        elif current_state == 'REJECTION_WARNINGS':
            if intent == 'ready_to_apply':
                return 'APPLICATION_HELP'
            elif intent == 'need_human_help':
                return 'CSC_HANDOFF'
            elif intent == 'ask_about_documents':
                return 'DOCUMENT_GUIDANCE'
            elif intent == 'go_back_to_scheme':
                return 'SCHEME_DETAILS'
        
        elif current_state == 'APPLICATION_HELP':
            if intent == 'need_human_help':
                return 'CSC_HANDOFF'
            elif intent == 'explore_other_schemes':
                return 'SCHEME_PRESENTATION'
            elif intent == 'ask_about_documents':
                return 'DOCUMENT_GUIDANCE'
        
        elif current_state == 'CSC_HANDOFF':
            if intent == 'explore_other_schemes':
                return 'SCHEME_PRESENTATION'
            elif intent == 'start_fresh':
                return 'GREETING'
        
        # Default: stay in current state
        return current_state
    
    def is_valid_transition(self, from_state: str, to_state: str) -> bool:
        """Check if state transition is valid."""
        return to_state in self.TRANSITIONS.get(from_state, [])


def has_minimum_profile(user_profile: UserProfile) -> bool:
    """
    Check if user profile has minimum required attributes for scheme matching.
    Validates: Requirements FR3 (Conversational Eligibility Profiling)
    
    Required attributes from Requirements FR3.4:
    - age, gender, annual_income, caste_category, domicile_status
    """
    required_attrs = ['age', 'gender', 'annual_income', 'category']
    
    for attr in required_attrs:
        value = getattr(user_profile, attr, None)
        if value is None:
            return False
    
    # Domicile is always 'delhi' for this system (Requirements UserProfile)
    return True


async def maybe_refresh_conversation_summary(session: Session) -> Session:
    """
    Refresh the conversation summary every 5 turns to preserve early context
    that falls outside the 10-message sliding window.
    
    Validates: Requirements FR9 (Context maintained across 20+ turns)
    """
    turn_count = len(session.messages)
    
    if turn_count > 0 and turn_count % 5 == 0:
        # Generate compressed summary from full message history + existing summary
        summary_prompt = f"""
        Previous summary: {session.conversation_summary or 'None'}
        Recent messages: {[m.content for m in session.messages]}
        
        Generate a concise summary (~200 tokens) covering:
        - User's life situation and stated needs
        - Profile attributes collected so far
        - Schemes discussed and user's reactions
        - Current stage in the process
        """
        
        session.conversation_summary = await llm_summarize(summary_prompt)
    
    return session
```

### 4. Voice Processing Pipeline

Integrates with Bhashini for Hindi voice processing with fallback mechanisms.

**Design Rationale:** Bhashini (AI4Bharat) provides government-approved, India-hosted AI services for Hindi. Fallbacks ensure graceful degradation if Bhashini is unavailable (Requirements NFR3).

```python
class VoiceProcessor:
    """
    Handles voice input/output using Bhashini APIs.
    Validates: Requirements FR1 (Voice-First Hindi Interface)
    """
    
    CONFIDENCE_THRESHOLD = 0.6  # Below this, ask user to repeat or switch to text
    
    async def process_voice_input(self, audio_data: bytes, session_language: Optional[str] = None) -> dict:
        """
        Convert voice to text using IndicWhisper.
        Target: >85% transcription accuracy (Requirements FR1.2)
        Latency: <5 seconds end-to-end (Requirements FR1.3)
        
        Returns dict with 'transcription', 'language', 'confidence'.
        """
        try:
            # Step 1: Detect language if not already known from session
            if session_language and session_language != 'auto':
                detected_language = session_language
            else:
                detected_language = await bhashini_client.detect_language(audio_data)
            
            # Step 2: Transcribe with detected language
            response = await bhashini_client.speech_to_text(
                audio_data=audio_data,
                language=detected_language,
                model='indicwhisper'
            )
            
            # Step 3: Check transcription confidence
            confidence = response.get('confidence', 1.0)
            if confidence < self.CONFIDENCE_THRESHOLD:
                return {
                    'transcription': None,
                    'language': detected_language,
                    'confidence': confidence,
                    'low_confidence': True,
                    'fallback_message': "मैं आपकी बात ठीक से समझ नहीं पाया। कृपया दोबारा बोलें या टेक्स्ट में लिखें।"
                }
            
            return {
                'transcription': response['transcription'],
                'language': detected_language,
                'confidence': confidence,
                'low_confidence': False
            }
            
        except BhashiniAPIError as e:
            logger.warning(f"Bhashini API failed: {e}, falling back to Google")
            # Fallback: Google Speech API (Requirements NFR3)
            return await fallback_speech_to_text(audio_data)
        
        except Exception as e:
            logger.error(f"Voice processing failed: {e}")
            # Ultimate fallback: Request text input (Requirements FR1.5)
            raise VoiceProcessingError("Please send text message instead")
    
    async def generate_voice_response(self, text: str, language: str = 'hi') -> str:
        """
        Convert text to voice using IndicTTS.
        Returns S3 URL for audio file.
        """
        try:
            # Primary: Call Bhashini IndicTTS API
            response = await bhashini_client.text_to_speech(
                text=text,
                language=language,
                model='indictts',
                voice='female'  # Default voice
            )
            
            # Upload to S3 and return URL
            audio_url = await upload_audio_to_s3(response['audio_data'])
            return audio_url
            
        except BhashiniAPIError as e:
            logger.warning(f"Bhashini TTS failed: {e}, falling back to Google")
            # Fallback: Google TTS
            return await fallback_text_to_speech(text, language)
        
        except Exception as e:
            logger.error(f"TTS generation failed: {e}")
            # Ultimate fallback: Text-only response
            return None  # Client will display text only


### 5. CSC Handoff Summary Generation

**Design Rationale:** When users need human help, CSC operators need complete context to assist effectively without re-asking questions (Requirements FR8.5).

```python
class CSCHandoffManager:
    """
    Manages handoff to Common Service Centres with context preservation.
    Validates: Requirements FR8 (CSC Locator and Human Handoff)
    """
    
    def generate_handoff_summary(self, session: Session) -> dict:
        """
        Generate comprehensive summary for CSC operator.
        """
        user_profile = session.user_profile
        eligible_schemes = session.context.get('eligible_schemes', [])
        discussed_schemes = session.discussed_schemes
        
        summary = {
            'user_info': {
                'phone': session.user_id,
                'language_preference': session.language_preference,
                'life_event': user_profile.life_event
            },
            'profile': {
                'age': user_profile.age,
                'gender': user_profile.gender,
                'category': user_profile.category,
                'annual_income': user_profile.annual_income,
                'employment_status': user_profile.employment_status,
                'district': user_profile.district
            },
            'eligible_schemes': [
                {
                    'id': scheme.id,
                    'name': scheme.name,
                    'name_hindi': scheme.name_hindi,
                    'benefits_amount': scheme.benefits_amount,
                    'documents_required': scheme.documents_required
                }
                for scheme in eligible_schemes[:5]
            ],
            'conversation_context': {
                'current_state': session.state,
                'discussed_schemes': discussed_schemes,
                'key_questions': self._extract_key_questions(session.messages)
            },
            'recommended_action': self._recommend_action(session)
        }
        
        return summary
    
    def _extract_key_questions(self, messages: List[Message]) -> List[str]:
        """Extract user's key questions from conversation history."""
        questions = []
        for msg in messages:
            if msg.sender == 'user' and '?' in msg.content:
                questions.append(msg.content)
        return questions[-3:]  # Last 3 questions
    
    def _recommend_action(self, session: Session) -> str:
        """Recommend next action for CSC operator."""
        state = session.state
        
        if state == 'SCHEME_MATCHING' and not session.context.get('eligible_schemes'):
            return "No eligible schemes found. Help user refine profile or explore alternative options."
        elif state == 'DOCUMENT_GUIDANCE':
            return "User needs help obtaining required documents. Guide through document procurement process."
        elif state == 'APPLICATION_HELP':
            return "User ready to apply. Assist with online/offline application submission."
        else:
            return "User needs general assistance with scheme navigation."
    
    async def find_nearest_csc(
        self, 
        district: str, 
        lat: Optional[float] = None, 
        lng: Optional[float] = None,
        limit: int = 5
    ) -> List[Office]:
        """
        Find nearest CSCs based on district or GPS coordinates.
        Validates: Requirements FR8.2 (Distance calculation within 500m accuracy)
        """
        if lat and lng:
            # GPS-based search using PostGIS
            cscs = await db.query("""
                SELECT *, 
                    ST_Distance(
                        ST_MakePoint(:lng, :lat)::geography,
                        ST_MakePoint(longitude, latitude)::geography
                    ) as distance
                FROM offices
                WHERE type = 'CSC' AND district = :district
                ORDER BY distance
                LIMIT :limit
            """, {'lat': lat, 'lng': lng, 'district': district, 'limit': limit})
        else:
            # District-based search
            cscs = await db.query("""
                SELECT * FROM offices
                WHERE type = 'CSC' AND district = :district
                ORDER BY name
                LIMIT :limit
            """, {'district': district, 'limit': limit})
        
        return cscs

    Client for AI4Bharat Bhashini APIs.

        """

            "audio": base64.b64encode(audio_data).decode(),

            "model": "indicwhisper"

        Convert text to speech using IndicTTS.

        payload = {
            "text": text,
            "language": language,
            "model": "indictts",
            "voice": "female"  # Default voice
        }
        
        response = await self.make_request("/text-to-speech", payload)
        return response
    
    async def translate(self, text: str, source_lang: str, target_lang: str) -> dict:
        """
        Translate text using IndicTrans2.
        
        Note: Translation is only used when necessary:
        - Hindi → English: When LLM requires English input (Claude/GPT-4)
        - English → Hindi: When generating Hindi responses from English LLM output

            "text": text,
            "source_language": source_lang,
            "target_language": target_lang,
            "model": "indictrans2"
        }
        
        response = await self.make_request("/translate", payload)

### Embedding Model Selection

**Decision:** Voyage AI voyage-multilingual-2 for scheme descriptions.

**Rationale:**
- Optimized for multilingual retrieval and RAG
- 1024 dimensions - efficient for scheme descriptions
- Excellent Hindi-English code-mixed text support
- 32,000 token context length for long scheme descriptions
- Meets <500ms retrieval latency requirement (Requirements FR4.5)

**Alternative Considered:** OpenAI text-embedding-3-small
- 1536 dimensions, good multilingual support
- Can be used as fallback if Voyage AI is unavailable

**Alternative Considered:** AWS Titan Embeddings
- Native AWS integration for Bedrock deployment
- 1024 dimensions, viable for full AWS-native stack

**Alternative Considered:** Sentence-BERT multilingual models
- Rejected due to lower quality for Hindi-English code-mixing
- Self-hosting adds operational complexity

**Alternative Considered:** Amazon Titan Embeddings (AWS-native)
- Viable option for full AWS-native stack
- Currently lower benchmark scores for Hindi-English code-mixed text compared to OpenAI
- Can be swapped in if multilingual performance improves, keeping the architecture unchanged

class EmbeddingService:
    """
    Manages text embeddings for semantic search.
    """
    
    def __init__(self):
        self.model = "text-embedding-3-small"
        self.dimension = 1536
        self.cache = {}  # In-memory cache for frequent queries
    
    async def embed_text(self, text: str, language: str = 'hi') -> List[float]:
        """
        Generate embedding for text.
        Caches embeddings for frequent queries.
        """
        # Check cache first
        cache_key = f"{language}:{text[:100]}"
        if cache_key in self.cache:
            return self.cache[cache_key]
        
        # Generate embedding
        response = await openai_client.embeddings.create(
            model=self.model,
            input=text
        )
        
        embedding = response.data[0].embedding
        
        # Cache for future use
        self.cache[cache_key] = embedding
        
        return embedding
    
    async def embed_scheme_descriptions(self, schemes: List[Scheme]):

            model=self.model,
            input=texts
        )

        ]
        
        for provider_name, client in providers:
            try:
                return await client.generate(prompt)
            except Exception as e:
                logger.warning(f"{provider_name} failed: {e}")
        
        # Ultimate fallback: Rule-based responses
        return generate_rule_based_response(prompt)
```

### Validation and Sanitization

```python
class InputValidator:
    """
    Validates and sanitizes user input.
    """
    
    def validate_age(self, age: int) -> bool:
        """Validate age is reasonable."""
        return 0 < age < 120
    
    def validate_income(self, income: int) -> bool:
        """Validate income is reasonable."""
        return 0 <= income < 100000000  # 10 crore max
    
    def sanitize_text(self, text: str) -> str:
        """
        Sanitize text input to prevent prompt injection.
        """
        # Remove potential prompt injection patterns
        dangerous_patterns = [
            r'ignore previous instructions',
            r'system:',
            r'<\|im_start\|>',
            r'<\|im_end\|>'
        ]
        
        for pattern in dangerous_patterns:
            text = re.sub(pattern, '', text, flags=re.IGNORECASE)
        
        # Limit length
        return text[:2000]
    
    def validate_audio(self, audio_data: bytes) -> bool:
        """
        Validate audio file.
        """
        # Check size (max 16MB for WhatsApp)
        if len(audio_data) > 16 * 1024 * 1024:
            raise AudioTooLargeError("Audio file too large")
        
        # Check format (basic magic number check)
        # OGG: 4F 67 67 53, MP3: FF FB or FF F3, WAV: 52 49 46 46
        valid_headers = [b'OggS', b'\xff\xfb', b'\xff\xf3', b'RIFF']
        
        if not any(audio_data.startswith(header) for header in valid_headers):
            raise InvalidAudioFormatError("Unsupported audio format")
        
        return True
```

## Performance Optimization

### Data Freshness and Verification

**Design Rationale:** Government scheme information changes frequently. Stale data leads to user frustration and failed applications (Requirements Risk Assessment).

**Verification Strategy:**
- **Monthly Sync:** Automated scraping of official government portals for scheme data refresh
- **User Feedback Flag:** Users can flag outdated or incorrect information directly in conversation
- **CSC Operator Reports:** CSC operators can report outdated information through a simple feedback interface
- **Last Verified Timestamp:** All schemes and documents track `last_verified` date
- **Staleness Warnings:** Display warning if data >90 days old, with link to official source

```python
class DataFreshnessManager:
    """
    Manages data freshness and verification.
    """
    
    def check_freshness(self, scheme: Scheme) -> dict:
        """
        Check if scheme data is fresh and return warnings.
        """
        days_since_verification = (datetime.now() - scheme.last_verified).days
        
        if days_since_verification > 90:
            return {
                'is_stale': True,
                'warning': 'This information was last verified {days} days ago. Please verify with official sources.',
                'official_url': scheme.official_url
            }
        
        return {'is_stale': False}
    
    async def refresh_scheme_data(self, scheme_id: str):
        """
        Refresh scheme data from official sources.
        """
        # Scrape official portal
        fresh_data = await scrape_official_portal(scheme_id)
        
        # Update database
        await db.update_scheme(scheme_id, fresh_data)
        
        # Update last_verified timestamp
        await db.update_timestamp(scheme_id)
```

### Caching Strategy
- **ElastiCache:** Scheme data (24-hour TTL), session data (24-hour TTL)
- **CloudFront CDN:** Static assets and audio files (edge caching)
- **Application Cache:** In-memory caching of embeddings

**Cache Hierarchy:**
1. **L1 - Lambda Memory:** Embedding cache, frequent queries (process lifetime)
2. **L2 - ElastiCache (Redis):** Scheme data (24-hour TTL, invalidated on DB update), session data (24-hour TTL matching DynamoDB)
3. **L3 - CloudFront CDN:** Audio files, static assets (7-day TTL, versioned URLs)

**Cache Invalidation:**
- Scheme data: Invalidate on database update (monthly refresh cycle)
- Audio files: Versioned URLs (no invalidation needed)
- Sessions: Automatic TTL expiry (24 hours)

**Note:** Scheme data and session data have different invalidation semantics despite sharing the same cache tier. Scheme data is invalidated on write (monthly); session data expires purely by TTL.
